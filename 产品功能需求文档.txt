## **《AI智能会议纪要生成助手》产品需求文档 (PRD)**

| **文档版本** | **V1.0** |
| :--- | :--- |
| **创建日期** | 2025年10月6日 |
| **产品名称** | AI智能会议纪要生成助手(MeetMemo) |

### 1. 产品概述

#### 1.1. 产品愿景
这是一款旨在将会议音频自动化、智能化地转化为结构清晰、重点突出的会议纪要的网页应用。通过解放与会者于繁琐的记录工作，提升会议效率，确保信息无遗漏，让每一次会议的价值都能沉淀和流转。

#### 1.2. 目标用户
- **项目经理、产品经理、团队负责人**：需要记录和分发会议决策、任务分配。
- **行政人员、秘书**：负责为管理层或多个团队整理正式的会议纪要。
- **学生、研究员**：需要整理课堂录音、学术研讨会、访谈记录。
- **任何需要将口头讨论转化为书面文字的个人或团队**。

#### 1.3. 核心问题 (Pain Points)
1.  **耗时耗力**：手动整理会议录音，既耗时又枯燥，且容易出错。
2.  **信息遗漏**：会议中信息量大，仅靠手动记录难以保证关键细节和决策的完整性。
3.  **效率低下**：会后需要花费大量时间回顾录音、整理笔记，导致会议结论和行动项（Action Items）延迟传达。
4.  **格式不一**：不同人整理的纪要格式五花八门，不利于团队内部的标准化管理和查阅。

#### 1.4. 解决方案
本产品通过一个简单的工作流解决上述问题：
1.  **上传音频**：用户上传会议录音文件。
2.  **AI转录**：系统后台调用**离线部署的OpenAI Whisper模型**将音频精准地转写为原始文字稿，确保音频数据不离开服务器，保障数据隐私。
3.  **AI整理**：系统将转录后的文字稿，结合用户选择的纪要模板，通过**DeepSeek API**进行智能分析、摘要、提炼和格式化，生成结构化的会议纪要初稿。
4.  **编辑与导出**：用户在网页上对AI生成的纪要进行最终的校对和修改，并以多种格式导出。

### 2. 功能需求 (Functional Requirements)

#### 2.1. 核心工作流：会议纪要生成

**FR-01: 音频文件上传**
- **描述**：用户能够上传本地的会议录音文件作为处理的起点。
- **功能点**：
    - 支持拖拽上传和点击选择文件两种方式。
    - 明确支持的音频格式，如 `.mp3`, `.wav`, `.m4a`, `.flac` 等。
    - 设置并提示文件大小上限（例如：500MB）。
    - 上传过程中显示清晰的进度条和上传速度。
    - 上传完成后给予用户明确的成功提示。

**FR-02: 处理参数配置**
- **描述**：在开始处理前，用户可以选择不同的处理模型和纪要模板，以获得最符合需求的结果。
- **功能点**：
    - **Whisper模型选择**：允许用户根据对速度和精度的不同要求选择Whisper模型（如 `base`, `small`, `medium`, `large`）。需在选项旁注明“速度快/精度一般”或“速度慢/精度最高”等提示。
    - **语言识别**：提供语言选择下拉框（如自动检测、中文、英文等），以提高转录准确率。
    - **DeepSeek API密钥配置**：用户需要在设置页面输入并保存自己的DeepSeek API Key。系统需对Key进行加密存储。
    - **纪要模板选择**：提供一个下拉菜单，让用户选择用于生成纪要的格式模板（详见FR-05）。

**FR-03: 任务处理与状态展示**
- **描述**：在文件上传和配置完成后，系统开始后台处理任务，并向用户实时反馈处理进度。
- **功能点**：
    - 点击“开始生成”按钮后，任务进入后台处理队列。
    - 前端界面清晰地展示当前任务状态，分为几个阶段：
        1.  **等待处理**：任务已提交，正在队列中排队。
        2.  **语音转录中...**：后台正在调用Whisper模型进行处理。此阶段可能较长，应提供预估剩余时间或动态提示。
        3.  **纪要生成中...**：正在调用DeepSeek API进行内容整理。
        4.  **处理完成**。
    - 用户在等待期间可以离开页面，处理完成后可通过邮件或浏览器通知（可选）提醒用户。

**FR-04: 结果展示与在线编辑**
- **描述**：处理完成后，在一个双栏对比视图中向用户展示结果，方便校对和修改。
- **功能点**：
    - **双栏视图**：
        - **左栏（原始记录）**：展示由Whisper生成的、带时间戳的原始文字稿。文字与时间戳关联，点击文字可跳转到对应音频时间点（高级功能）。
        - **右栏（会议纪要）**：展示由DeepSeek API根据模板生成的结构化会议纪要。
    - **富文本编辑器**：右栏的会议纪要应在一个所见即所得（WYSIWYG）的编辑器中，支持用户进行二次修改，如调整文本、修正错别字、补充内容等。
    - **内置音频播放器**：页面顶部或侧边提供一个音频播放器，用户可以随时播放原始录音，方便对照原文进行核对。

**FR-05: 纪要模板管理**
- **描述**：用户可以自定义会议纪要的模板，以适应不同的会议类型和团队规范。
- **功能点**：
    - **预设模板**：系统内置几种常用模板，如“标准会议纪要”、“决策会议模板”、“头脑风暴模板”等。
    - **自定义模板**：
        - 用户可以创建新的模板。
        - 模板编辑器允许用户通过简单的文本输入和占位符（如 `{原始文本}`）来定义纪要的结构。例如：
          ```markdown
          # 会议纪要

          **会议主题**：[请根据{原始文本}总结会议的核心主题]
          **参会人员**：[请根据{原始文本}列出参会人员]
          **主要议题与讨论**：[请详细分点总结{原始文本}中的主要议题和各方观点]
          **决议事项**：[请提炼{原始文本}中形成的所有明确决议]
          **行动项 (Action Items)**：
          - [负责人] - [任务描述] - [截止日期]
          ```
    - 用户可以对自己创建的模板进行保存、编辑和删除。
    - 可以设置一个模板为“默认模板”。

**FR-06: 导出与分享**
- **描述**：用户可以将最终确认的会议纪要导出为多种格式。
- **功能点**：
    - **一键复制**：提供“复制到剪贴板”按钮。
    - **文件导出**：支持导出为 `.md` (Markdown), `.txt` (纯文本), `.docx` (Word文档) 格式。

#### 2.2. 用户与历史记录管理

**FR-07: 用户账户系统 (可选，建议MVP后加入)**
- **描述**：提供用户注册和登录功能，以保存个人数据。
- **功能点**：
    - 支持邮箱/手机号注册和登录。
    - 提供密码找回功能。

**FR-08: 历史记录管理**
- **描述**：用户可以查看、管理自己处理过的所有会议纪要。
- **功能点**：
    - 提供一个历史记录列表，展示每次任务的名称、上传时间、处理状态等。
    - 用户可以点击进入任一历史记录，重新查看或编辑当时的转录稿和会议纪要。
    - 提供对历史记录的搜索和删除功能。

### 3. 非功能性需求 (Non-Functional Requirements)

| 类别 | 需求描述 |
| :--- | :--- |
| **性能** | - **Whisper处理**：应明确告知用户，根据音频时长和所选模型，转录过程可能需要数分钟到数十分钟。系统应能稳定处理长达2小时的音频文件。 <br> - **API响应**：DeepSeek API的调用和响应时间应在30秒内完成（取决于文本长度）。 <br> - **页面加载**：主要页面的加载时间应在3秒以内。 |
| **安全性** | - **数据隐私**：**核心需求**。明确向用户声明，音频文件在服务器端处理，**绝不会**发送给任何第三方。只有转录后的文本会发送给DeepSeek API。 <br> - **API Key安全**：用户提交的DeepSeek API Key必须在数据库中加密存储，在传输过程中使用HTTPS。 <br> - **访问控制**：用户只能访问自己的历史记录。 |
| **可用性** | - **UI/UX**：界面设计应简洁、直观，工作流引导清晰，让非技术用户也能轻松上手。 <br> - **错误处理**：对于上传失败、格式不支持、API调用失败等异常情况，应向用户提供友好且明确的错误提示。 |
| **可扩展性** | - **异步任务处理**：后端必须使用任务队列（如Celery, RQ）来处理耗时的Whisper转录任务，以防HTTP请求超时和服务器阻塞，保证系统能应对并发请求。 |

### 4. 技术栈与架构考量

- **前端**：建议使用现代JavaScript框架，如 `React`, `Vue` 或 `Svelte`，以便构建交互性强的单页应用。
- **后端**：`Python` 是最佳选择，因为它能无缝集成 `whisper` 库。Web框架可选用 `FastAPI` 或 `Flask`。
- **Whisper部署**：在后端服务器上直接部署开源的Whisper模型。使用带NVIDIA GPU的服务器。
- **DeepSeek API**：通过标准的HTTP请求调用其服务。
- **数据库**：使用 `SQLite` 或 `MySQL` 存储用户信息、任务历史、模板等。
- **任务队列**：使用 `Celery` + `Redis`/`RabbitMQ` 来管理和执行异步的音频处理任务。
- **部署**：建议使用 `Docker` 进行容器化部署，简化环境配置和依赖管理。

---

### 5. MVP (最小可行产品) 规划

为了快速验证核心价值和市场反馈，MVP应聚焦于最核心的功能：

1.  **实现核心工作流**：支持音频上传 -> 选择固定的Whisper模型（如`base`） -> 调用DeepSeek API -> 在一个简单页面上展示原始文本和生成的纪要。
2.  **简化配置**：暂时不支持多模型选择。API Key直接在页面输入，暂不保存。
3.  **固定模板**：内置1个效果最好的会议纪要模板，暂不支持自定义。
4.  **简化输出**：只提供“复制到剪贴板”功能。
5.  **无用户系统**：无需登录，所有操作基于会话（Session）。

通过MVP，可以快速验证“Whisper离线转录 + DeepSeek API整理”这一技术路径的可行性和最终效果，为后续迭代收集宝贵的用户反馈。
